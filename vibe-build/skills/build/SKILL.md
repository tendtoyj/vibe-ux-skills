# 개요

이 스킬은 마일스톤 하나를 통째로 구현한다.

예를 들어 `/build M1`로 호출되면 다음 파일을 읽는다.

```text
docs/milestones/M1/plan.md
docs/milestones/M1/tasks.md
```

그 다음 `tasks.md`에서 아직 완료되지 않은 가장 첫 태스크부터 시작해, 모든 태스크가 완료될 때까지 멈추지 않고 진행한다.

기본 실행 루프는 다음과 같다.

```text
RED -> GREEN -> build/verify -> commit -> mark complete -> next task
```

# 입력

사용자는 반드시 마일스톤을 지정해야 한다.

허용 예시:

- `/build M1`
- `/build m2`
- `/build Milestone 3`
- `/build 4`

마일스톤이 명시되지 않으면 실행하지 말고 어떤 마일스톤을 빌드할지 물어본다.

# 필수 파일

대상 마일스톤 번호를 `M<N>`으로 정규화한 뒤 다음 파일을 읽는다.

1. `docs/milestones/M<N>/plan.md`
2. `docs/milestones/M<N>/tasks.md`

필요하면 다음 상위 문서도 읽어 요구사항과 범위를 확인한다.

1. `docs/PRD.md`
2. `docs/ROADMAP.md`
3. `docs/DESIGN.md`

`plan.md` 또는 `tasks.md`가 없으면 구현하지 말고, 먼저 해당 마일스톤의 plan 스킬을 실행해야 한다고 보고한다.

# 실행 범위

이 스킬은 지정된 마일스톤의 `tasks.md`에 있는 미완료 태스크만 구현한다.

하지 말아야 할 것:

- 다른 마일스톤 태스크 구현
- `tasks.md`에 없는 기능 추가
- 인접 코드 정리나 리팩터링을 핑계로 범위 확장
- 실패한 테스트 삭제
- unrelated local change를 커밋에 포함

작업 중 관련 문제가 보이면 구현하지 말고 메모만 남긴다.

# 실행 전 점검

1. `git status --short`를 확인한다.
2. 기존 변경사항이 있으면, 이번 build가 만든 변경인지 아닌지 구분한다.
3. 관련 없는 변경사항은 건드리거나 커밋하지 않는다.
4. `tasks.md`에서 완료 표시된 태스크와 미완료 태스크를 파악한다.
5. 첫 번째 미완료 태스크부터 시작한다.

완료 태스크 인식 기준:

- 완료: `- [x]`, `- [X]` 또는 태스크 제목/본문에 명확한 완료 표시
- 미완료: `- [ ]`, 또는 완료 표시가 없는 `Task T###` 섹션

# 디자인 토큰을 Tailwind에 매핑할 때 (M1 스타일 스캐폴딩)

M1에서 `docs/DESIGN.md`의 디자인 토큰을 Tailwind v4 `@theme`로 옮길 때 아래를 지킨다.

Tailwind v4의 `--spacing-*` 네임스페이스는 여백(padding/margin/gap)뿐 아니라
크기 유틸리티(`w-`, `h-`, `size-`, `max-w-`, `min-w-`, `max-h-`, `min-h-`,
`inset-`, `top/right/bottom/left-`, `translate-`)까지 함께 읽는다.
따라서 DESIGN.md의 t셔츠 간격 이름(`sm`, `md`, `lg`, `xl`…)을 `--spacing-*`에
그대로 정의하면, `max-w-md` 같은 유틸이 같은 이름을 끌어다 써서 조용히 깨진다
(`max-w-md`가 컨테이너 폭 대신 간격 값 12px으로 해석됨).

규칙:

1. 여백 토큰은 `--spacing-*`에만 둔다. `gap-md`, `px-base`, `py-xxl`, `mt-lg`는
   의도된 용도이므로 정상.
2. 콘텐츠 폭·크기에는 t셔츠 키워드 유틸을 쓰지 않는다. `max-w-md`, `min-w-sm`,
   `max-h-lg` 금지.
   - 기본: 임의값으로 쓴다. 예: `max-w-[28rem]`, `max-w-[36rem]`.
   - 같은 폭이 여러 곳에서 반복되면 `@theme`에 컨테이너 전용 토큰
     (`--container-md: 28rem`)을 정의해 의미 이름을 살린다. (spacing의 `md`와
     container의 `md`는 별도 개념임을 주석으로 남긴다.)
3. 충돌이 의심되면 컴파일된 CSS로 검증한다. 빌드 산출 CSS에서 `max-w-md`가
   `var(--spacing-*)`로 해석되면 충돌난 것이다.

여백 계열(`p-`, `m-`, `gap-`, `space-`)은 `--spacing-*`의 원래 목적이므로 안전하다.

# 태스크 실행 루프

각 태스크마다 아래 순서를 지킨다.

## 1. 태스크 이해

해당 태스크의 다음 항목을 읽는다.

- Description
- Acceptance Criteria
- Verification
- Dependencies
- Likely Files
- Estimated Scope

의존 태스크가 미완료라면 현재 태스크를 진행하지 말고 의존 태스크부터 처리한다.

## 2. 관련 코드 읽기

코드를 쓰기 전에 관련 파일과 기존 패턴을 읽는다.

- 기존 컴포넌트/서비스/테스트 패턴
- 라우팅/상태 관리 방식
- 스타일링 방식
- 테스트 명령과 빌드 명령

추측으로 새 구조를 만들지 말고 기존 프로젝트 패턴을 우선한다.

## 3. RED

먼저 실패하는 테스트 또는 실패하는 검증 조건을 만든다.

자세한 테스트 원칙은 `tdd.md`를 따른다.

기본 원칙:

- 새 동작이나 버그 수정은 먼저 실패 테스트로 증명한다.
- 테스트는 구현 세부가 아니라 public behavior를 검증한다.
- 한 번에 하나의 행동만 검증한다.
- 테스트를 대량으로 먼저 만들지 않는다.

순수 디자인 목업, 정적 UI, 문서/설정처럼 자동 테스트가 부적절한 태스크라면 다음 중 가장 강한 검증을 RED로 삼는다.

1. 컴포넌트 렌더링/접근성/상태 테스트
2. 타입 체크 또는 빌드에서 실패하는 최소 조건
3. 명시적인 수동 검증 체크리스트

자동 테스트를 추가하지 않는 경우, 태스크 완료 요약에 이유를 짧게 남긴다.

## 4. GREEN

현재 태스크의 Acceptance Criteria를 만족하는 최소 구현을 한다.

- 현재 태스크에 필요한 코드만 작성한다.
- 미래 태스크를 예상해 추상화하지 않는다.
- 세 번째 반복 전까지는 중복보다 명확성을 우선한다.
- 새 dependency 추가, 스키마 변경, 인증/권한, 결제, destructive migration처럼 위험한 작업은 태스크에 명시되어 있지 않으면 중단하고 사용자 확인을 받는다.

## 5. Build / Verify

태스크의 Verification에 적힌 명령을 실행한다.

명령이 명시되어 있지 않으면 프로젝트에서 확인 가능한 가장 적절한 검증을 실행한다.

우선순위:

1. 해당 태스크 테스트
2. 전체 테스트
3. 타입 체크
4. lint
5. build
6. 수동 검증

검증 실패 시 실패 원인을 고치고 다시 검증한다. 실패 상태로 커밋하거나 완료 표시하지 않는다.

## 6. Commit

검증이 통과하면 태스크 단위로 커밋한다.

커밋 원칙:

- 한 태스크 = 한 커밋을 기본으로 한다.
- 커밋은 독립적으로 되돌릴 수 있어야 한다.
- `git add -A`를 쓰지 않는다.
- 이번 태스크에서 의도적으로 변경한 파일과 `tasks.md` 완료 표시만 stage한다.
- unrelated local change는 포함하지 않는다.
- 커밋 메시지는 마일스톤과 태스크 ID를 포함한다.

권장 형식:

```text
feat(M<N>): complete T### <짧은 설명>
```

기능이 아닌 목업/문서/설정이면 `feat`, `chore`, `test`, `docs` 중 맞는 타입을 사용한다.

## 7. Mark Complete

커밋 후 `docs/milestones/M<N>/tasks.md`에서 해당 태스크를 완료 표시한다.

- 체크박스가 있으면 `[ ]`를 `[x]`로 바꾼다.
- 체크박스가 없는 `Task T###` 형식이면 제목 옆이나 태스크 하단에 완료 표시를 추가한다.
- 가능하면 완료 요약에 검증 명령과 커밋 SHA를 짧게 남긴다.

완료 표시 변경도 해당 태스크 커밋에 포함하는 것이 원칙이다. 만약 커밋 후에 완료 표시를 했다면 즉시 별도 `docs`/`chore` 커밋으로 남기지 말고, 가능한 경우 같은 태스크 커밋에 amend한다. amend가 위험하면 다음 커밋에 섞지 말고 사용자에게 보고한다.

## 8. 다음 태스크

다음 미완료 태스크로 이동한다.

사용자 확인 없이 계속 진행한다. 단, 아래 중 하나가 발생하면 멈춘다.

# 중단 조건

다음 상황에서는 멈추고 사용자에게 보고한다.

- `plan.md` 또는 `tasks.md`가 없음
- 태스크 요구사항이 모순되거나 PRD/ROADMAP과 충돌
- 테스트나 빌드 실패를 합리적으로 해결할 수 없음
- 태스크 수행에 사용자 결정이 필요함
- destructive migration, 결제, 권한/보안, 외부 서비스 비용 발생 등 되돌리기 어려운 작업이 필요함
- unrelated local change와 충돌해 안전하게 커밋할 수 없음
- 마일스톤 범위를 벗어난 작업이 필요함

# 완료 기준

마일스톤 build는 다음을 모두 만족해야 완료다.

- [ ] 대상 `tasks.md`의 모든 태스크가 완료 표시됨
- [ ] 각 태스크가 Acceptance Criteria를 충족함
- [ ] 각 태스크의 Verification이 통과함
- [ ] 각 태스크 또는 논리적 단위별 커밋이 존재함
- [ ] 마지막 전체 build/test 검증이 통과함
- [ ] 마일스톤 범위를 벗어난 변경이 없음
- [ ] 필요한 데이터베이스 마이그레이션이 모두 완료됨

# 최종 응답

완료 후 다음을 간단히 보고한다.

- 완료한 마일스톤
- 완료한 태스크 수
- 실행한 주요 검증 명령
- 생성한 커밋 목록
- 남은 작업 또는 중단 사유가 있다면 그 내용
