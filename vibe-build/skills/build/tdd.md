# TDD Reference

이 문서는 `build` 스킬에서 태스크를 구현할 때 따르는 간결한 테스트 주도 개발 기준이다.

## 핵심 원칙

- 테스트는 구현 세부가 아니라 사용자가 관찰할 수 있는 동작을 검증한다.
- 가능한 한 public interface를 통해 검증한다.
- 한 번에 하나의 테스트, 하나의 행동만 다룬다.
- 테스트를 전부 먼저 만들고 구현을 나중에 몰아서 하지 않는다.
- RED 상태에서 리팩터링하지 않는다.
- Frontend UI 코드의 경우 TDD를 적용하지 않는다.

## RED -> GREEN -> REFACTOR

1. **RED**: 현재 태스크의 가장 중요한 행동 하나를 검증하는 실패 테스트를 작성한다.
2. **GREEN**: 그 테스트를 통과시키는 최소 구현을 한다.
3. **REFACTOR**: 테스트가 통과한 상태에서만 이름, 중복, 구조를 정리한다.
4. 다음 행동으로 반복한다.

## 좋은 테스트

- “무엇이 되어야 하는가”를 설명한다.
- 내부 함수 호출 여부보다 결과 상태, 화면, 응답, 저장 결과를 확인한다.
- 리팩터링해도 동작이 같으면 깨지지 않는다.
- 실패했을 때 어떤 요구사항이 깨졌는지 바로 알 수 있다.

## 나쁜 테스트

- private 함수나 내부 호출 순서를 검증한다.
- 실제 동작이 깨져도 통과한다.
- 리팩터링만 해도 깨진다.
- 너무 많은 행동을 한 테스트에 묶는다.

## Mocking 기준

선호 순서:

1. 실제 구현
2. in-memory fake
3. stub
4. interaction mock

mock은 외부 API, 이메일, 결제, 시간, 네트워크처럼 느리거나 비결정적이거나 부작용이 큰 경계에서만 사용한다.

## 목업/정적 UI 태스크

프론트엔드 목업처럼 자동 테스트가 애매한 태스크도 검증 없이 완료하지 않는다.

가능하면 다음 중 하나를 사용한다.

- 컴포넌트 렌더링 테스트
- 접근성/역할 기반 쿼리 테스트
- 주요 상태별 story나 fixture
- 타입 체크
- lint
- build
- 수동 검증 체크리스트

수동 검증만 가능한 경우, `tasks.md` 완료 요약에 무엇을 확인했는지 남긴다.

## 완료 전 확인

- [ ] 새 동작은 테스트 또는 명시적 검증으로 증명됨
- [ ] 기존 테스트가 통과함
- [ ] build/typecheck/lint 중 프로젝트에 맞는 검증이 통과함
- [ ] 테스트가 구현 세부에 과하게 결합되지 않음
- [ ] 현재 태스크 범위 밖의 기능을 추가하지 않음
