# 개요

이 스킬은 PRD 이후 바로 구현 로드맵으로 가지 않고, 프론트엔드 목업 스프린트에서 그릴 화면들을 먼저 계획한다.

코드를 작성하지 않는다. 산출물은 화면 단위 목업 계획 문서다.

# 입력

반드시 다음 파일을 읽는다.

1. `docs/PRD.md`
2. `docs/userflow.md`

둘 중 하나가 없거나 비어 있으면 문서를 작성하지 말고, 누락된 파일을 보고한다.

프로덕트 이해가 부족하면 다음 파일을 추가로 읽는다.

1. `docs/idea.md`
2. `docs/DESIGN.md`

`docs/DESIGN.md`가 있으면 시각 방향, 톤, 레이아웃 원칙, 컴포넌트 원칙을 화면 설계에 반영한다.

# 산출물

아래 두 파일을 작성한다.

```text
docs/mockups/plan.md
docs/mockups/screens.md
```

문서 구조는 반드시 이 스킬의 템플릿을 따른다.

1. `assets/plan-template.md`
2. `assets/screens-template.md`

템플릿을 먼저 읽고, 섹션 이름과 반복 구조를 유지한 채 프로젝트 내용으로 채운다.

# 작성 원칙

## 1. 화면 중심으로 쪼개기

목업 계획은 개발 태스크가 아니라 사용자가 리뷰할 수 있는 화면 단위로 나눈다.

좋은 screen:

- 하나의 명확한 사용자 목적을 가진다.
- PRD의 사용자 시나리오 또는 userflows의 한 순간을 눈으로 확인할 수 있다.
- 독립적으로 리뷰하고 수정할 수 있다.
- 이후 구현 시 하나의 목업 작업 단위로 체크할 수 있다.

나쁜 screen:

- "공통 컴포넌트 만들기"처럼 사용자가 직접 볼 수 없는 내부 작업
- "전체 UI"처럼 너무 큰 묶음
- 화면 상태를 전혀 고려하지 않는 단일 정적 페이지

## 2. 흐름 먼저, 상세는 나중

먼저 `docs/mockups/plan.md`에서 전체 화면 목록과 화면 간 관계를 설계한다.

그 다음 `docs/mockups/screens.md`에서 각 화면의 목적, 레이아웃, 상태, 리뷰 질문을 상세화한다.

## 3. 체크 가능한 화면 섹션

`plan.md`의 각 화면 섹션 제목은 반드시 체크박스를 포함한다.

형식:

```markdown
## [ ] Screen S001 - [화면 이름]
```

나중에 해당 화면 목업이 구현되고 리뷰를 통과하면 `[x]`로 바꿀 수 있어야 한다.

## 4. 화면 ID 유지

화면 ID는 `S001`, `S002`, `S003`처럼 부여한다.

같은 화면은 `plan.md`와 `screens.md`에서 동일한 ID를 사용한다.

화면 ID는 사용자 흐름 순서를 기준으로 배치한다. 보조 화면이나 상태 화면은 관련 화면 바로 뒤에 둔다.

## 5. ASCII 도식 필수

`screens.md`의 모든 개별 screen 상세에는 ASCII 도식이 있어야 한다.

도식은 정교한 와이어프레임이 아니라 화면의 정보 구조를 빠르게 이해하기 위한 것이다.

예시:

```text
 ┌───────────────────────────┐
   ▓ ▢ ▢ ▢ ▢ ▢
   매일 한 칸씩, 사진으로
                  [ 시작하기 ]
   로그인 없이 바로 시작
 └───────────────────────────┘
```

## 6. 목업 스프린트에 맞는 깊이

너무 깊은 구현 계획을 만들지 않는다.

포함할 것:

- 화면 목적
- 핵심 사용자 행동
- 화면 간 이동
- 주요 UI 영역
- 필요한 상태
- 리뷰 질문
- 확정 기준

제외할 것:

- 컴포넌트 내부 구현 방식
- API 설계
- DB 스키마
- 실제 인증/권한 로직
- 세부 CSS 코드
- 장기 구현 로드맵

# 작업 절차

1. `docs/PRD.md`와 `docs/userflows.md`를 읽는다.
2. 필요하면 `docs/idea.md`, `docs/DESIGN.md`를 읽는다.
3. PRD의 핵심 사용자 시나리오와 userflows의 단계들을 화면 후보로 변환한다.
4. 화면을 사용자 흐름 순서대로 정렬한다.
5. 중복 화면을 합치고, 상태만 다른 경우 같은 screen의 상태로 넣을지 별도 screen으로 둘지 결정한다.
6. `assets/plan-template.md`를 따라 `docs/mockups/plan.md`를 작성한다.
7. `assets/screens-template.md`를 따라 `docs/mockups/screens.md`를 작성한다.
8. 두 문서의 화면 ID, 화면명, 흐름 관계가 서로 일치하는지 확인한다.

# 최종 확인

작업을 마치기 전에 다음을 확인한다.

- [ ] `docs/PRD.md`와 `docs/userflows.md`를 모두 반영했다.
- [ ] 필요한 경우 `docs/idea.md`, `docs/DESIGN.md`를 추가 반영했다.
- [ ] `docs/mockups/plan.md`가 화면 목록과 화면 간 관계를 포함한다.
- [ ] `plan.md`의 각 화면 섹션 제목이 `## [ ] Screen S### - ...` 형식이다.
- [ ] `docs/mockups/screens.md`가 모든 화면의 상세 설계를 포함한다.
- [ ] `screens.md`의 모든 화면 상세에 ASCII 도식이 있다.
- [ ] 두 문서의 화면 ID가 일치한다.
- [ ] 백엔드, DB, API, 실제 기능 구현 계획으로 과도하게 내려가지 않았다.
