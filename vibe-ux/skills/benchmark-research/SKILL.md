# Benchmark Research

기획 중인 아이디어와 닮은 **경쟁 서비스·참고 프로덕트**를 조사해, 우리가 무엇을 다르게 가져갈지 결정하기 위한 리서치다. 깊은 시장 분석이 목적이 아니다 — 바이브 코딩으로 빠르게 만들 신규 서비스가 **어디에 자리 잡고, 무엇으로 차별화할지**를 잡는 가벼운 리서치다.

이 스킬은 Vibe UX 기획 플로우의 **두 번째 단계**다. `docs/idea.md`를 입력으로 받고, `docs/benchmark.md`(+ `docs/images/`)를 산출한다.

## 입력 / 산출물

- **입력:** `docs/idea.md` (brainstorming-idea 산출물). 없으면 "먼저 brainstorming-idea로 아이디어를 잡아주세요"라고 안내하거나, 사용자에게 직접 컨셉을 물어 진행한다.
- **산출물:** `docs/benchmark.md` + 다운로드한 참고 이미지 `docs/images/`.
- 별도의 리서치 메모리 폴더는 만들지 않는다. 이 문서 하나로 끝낸다.

---

## 원칙

- **깊이보다 방향.** 경쟁사 5개를 완벽 분석하는 게 아니라, 3~5개를 보고 **우리의 빈틈(차별 지점)** 을 찾는 게 목표다.
- **1차 정보를 본다.** 검색 스니펫만 믿지 말고, 핵심 후보는 공식 사이트를 사용 가능한 웹/브라우징 도구로 직접 확인한다.
- **출처는 반드시 남긴다.** 모든 주장과 이미지에 출처 URL을 붙인다. 모르면 모른다고 쓰고 추측으로 빈칸을 메우지 않는다.
- 기본 언어는 한국어. 표의 컬럼명은 영어로 둬도 좋지만 본문·설명은 한국어로 쓴다.

## 작업 흐름

### 1. idea.md를 읽고 검색 축을 정한다 — "어디서 무엇을 검색할 것인가"

`docs/idea.md`의 한 줄 컨셉·핵심 솔루션·타깃을 읽고, **무엇을 찾을지** 먼저 정한다. 막연히 "비슷한 앱"을 검색하지 말고, 다음 세 축으로 나눠 검색어를 설계한다:

- **직접 경쟁 (Direct):** 같은 문제를 같은 방식으로 푸는 서비스. ("OO 기록 앱", "OO 추천 서비스")
- **간접 경쟁 / 대체재 (Indirect):** 같은 문제를 다른 방식으로 푸는 것. 사용자가 지금 대신 쓰는 것(엑셀, 메모앱, 오프라인 방법 등).
- **참고 레퍼런스 (Reference):** 도메인이 달라도 UX·기능·플로우가 배울 만한 서비스.

각 축마다 검색어를 1~2개씩 만들어, 사용 가능한 웹 검색 도구를 **여러 각도로 3~5회** 쓴다. 한 번의 검색은 한 가지 프레임만 주므로 각도를 바꿔야 빈틈이 메워진다. 영어 검색어도 함께 쓰면 글로벌 레퍼런스가 잡힌다.

### 2. 후보를 추리고 1차 출처를 확인한다

검색 결과에서 직접 경쟁 3~5개 + 간접/참고 2~3개를 추린다. 핵심 후보는 공식 사이트·소개 페이지를 **직접 읽어** 포지셔닝·핵심 기능·가격·타깃을 정확히 파악한다. 웹 페이지를 읽는 도구가 페이지 안의 이미지(스크린샷/로고/대표 이미지) URL을 추출할 수 있으면 다음 단계 이미지 수집이 쉬워진다.

### 3. 참고 이미지를 수집한다 — "정보를 어떻게 가공할 것인가" (1)

각 서비스의 대표 화면·핵심 기능 스크린샷·랜딩 페이지 이미지를 모은다. `scripts/fetch_images.sh`로 다운로드하면 순번대로 저장하고, 실제 파일 타입을 확인해 가짜 이미지를 걸러낸다:

```bash
bash <스킬경로>/scripts/fetch_images.sh docs/images "URL1" "URL2" "URL3"
```

받은 이미지 중 화면 구성이 중요한 것은 사용 가능한 이미지 확인 도구로 직접 열어 **무엇이 인상적인지**(레이아웃, 핵심 동선, 카피)를 메모해 둔다. 이 메모가 benchmark.md의 관찰로 들어간다. 모든 이미지는 출처 URL과 함께 기록한다.

### 4. benchmark.md를 작성한다 — "정보를 어떻게 가공할 것인가" (2)

조사한 내용을 아래 구조로 정리한다. 핵심은 **나열이 아니라 비교**다. 표로 한눈에 비교하고, 마지막에 반드시 **우리의 차별 프레이밍**을 도출한다 — 이 리서치의 진짜 목적이다.

여기까지가 **기본 리서치**다. benchmark.md 초안이 만들어지면, 보고 전에 심화 여부를 묻는다.

### 4.5 심화 리서치 (선택)

기본 조사는 일반 웹 검색/페이지 확인 기준이라 사이트 문구·가격이 부정확하거나, 실제 화면을 못 볼 수 있다. 더 깊이 파고 싶은지 사용자에게 묻는다:

> "여기까지가 기본 조사예요. 경쟁사 사이트를 더 깊이 파볼까요? 사이트 문구·가격을 정확히 긁어오거나(Firecrawl), 실제 화면을 캡처해서 디자인까지 보는(Playwright) 심화 조사예요. 시간이 더 걸립니다."

**사용자가 원하면**, 필요한 MCP나 브라우저 도구가 연결돼 있는지 먼저 확인한다. 그리고 해당 도구로 보강한다:

- **Firecrawl 계열 스크래퍼:** 경쟁사 홈/가격 페이지를 스크랩해 **정확한 헤드라인·서브카피·CTA·가격 모델**을 가져온다. → `핵심 기능 비교`·`관찰`을 추측이 아니라 실측으로 채운다.
- **Playwright/Browser 계열 브라우저 도구:** 경쟁사 URL로 들어가 **실제 화면을 스크린샷**한다. 스크린샷은 `docs/images/`에 저장하고, 색·레이아웃·핵심 동선 같은 **디자인 관찰**을 추가한다.

둘 다 할 수도, 하나만 할 수도 있다. 심화로 채운 내용은 benchmark.md의 **기존 섹션을 보강**하고, 해당 항목에 `(심화)` 표시를 단다(아래 템플릿 참고).

**안전장치:**
- 필요한 MCP가 연결돼 있지 **않으면**, 그 사실을 사용자에게 알리고 **기본 리서치 결과로 마무리**한다. 없는 도구를 있는 척하거나 결과를 지어내지 않는다.
- 심화를 건너뛰어도 benchmark.md는 그 자체로 **완결**이다. 심화는 어디까지나 부가다.
- 스크랩·캡처한 정보도 **출처 URL을 반드시 남긴다.**

### 5. 검증하고 보고한다

폴더 구조를 확인하고(`find docs -type f`), 사용자에게 짧게 보고한다: 만든 문서 경로, 비교한 서비스 수, 받은 이미지 수, **심화 리서치 수행 여부**, 그리고 **우리가 잡은 차별 프레이밍 한 줄**. 확인이 더 필요한 불확실한 정보(2차 출처에만 있는 가격 등)는 따로 짚는다.

## 산출물 구조 — docs/benchmark.md (필수 산출물)

**이 스킬의 최종 산출물은 항상 `docs/benchmark.md` 파일이다.** 조사 결과를 채팅에만 보여주고 끝내지 않고, "파일로 저장할까요?"라고 **묻지 않는다** — 저장은 옵션이 아니라 결과물 그 자체다. 내용이 정리되면 파일로 **바로 저장한다.** (확인이 필요하면 *저장 여부*가 아니라 *내용*만 확인한다.) `docs/` 폴더가 없으면 만든다.

아래 구조를 따른다. 표는 채울 수 있는 만큼만 채우고, 모르는 칸은 "—"로 둔다.

```markdown

> Vibe UX · benchmark-research 산출물
```

## 사실성 원칙
- 가격·수치·고유명사는 1차 출처 기준. 2차 매체에만 있으면 "별도 확인 필요"로 표기한다.
- 출처 없는 주장은 쓰지 않는다.
- 모르면 모른다고 쓴다. 빈칸을 그럴듯한 추측으로 메우지 않는다.
