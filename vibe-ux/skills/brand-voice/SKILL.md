# Brand Voice

서비스의 **이름을 확정하고**, 그 브랜드가 어떻게 말하고 어떤 성격을 가지는지(보이스·톤·무드)를 정의해 문서로 남긴다. 산출물 `docs/brandvoice.md`는 이후 PRD의 "브랜드 & 톤" 섹션이자, 바이브 코딩 단계에서 UI 카피와 화면 분위기를 정하는 기준이 된다.

이 스킬은 Vibe UX 기획 플로우에서 **PRD 직전 단계**다.

## 입력 / 산출물

- **입력:** `docs/idea.md`(컨셉·타깃·이름 후보), `docs/benchmark.md`(경쟁의 톤, 차별 프레이밍). 있으면 읽어서 맥락으로 쓴다. 없어도 진행 가능하다.
- **산출물:** `docs/brandvoice.md`.

### 시작 시
`docs/brandvoice.md`가 이미 있으면 요약을 보여주고 "기존 것을 업데이트할까요, 새로 만들까요?"라고 묻는다. 없으면 새로 만든다.

---

## Step 0 — 서비스 이름 확정

브랜드 보이스를 정의하기 전에, idea.md에서 가칭으로 두었던 **이름을 확정**한다.

1. `docs/idea.md`의 "이름 후보 & 가칭"을 읽어 현재 후보를 보여준다.
2. 후보가 약하거나 더 보고 싶어 하면, 컨셉·타깃·차별 프레이밍에 맞는 이름 후보를 **3~5개 더 제안**한다. (부르기 쉬움, 의미 연결, 도메인/검색 중복 위험을 한 줄씩 곁들인다.)
3. 사용자가 한 이름으로 기울면, **사용 가능한 웹 검색 도구로 가볍게 검증한다.** 같은 이름의 서비스·앱이 이미 있는지, 명백한 상표나 검색 중복이 있는지 확인한다.
   - **반드시 타깃 시장의 언어·스토어 기준으로 검색한다.** 검색 도구의 지역·언어 기본값 때문에 한국 대상 서비스를 영어로만 검색하면 동일 이름이 있어도 못 잡을 수 있다. 한국 서비스라면 **이름을 한국어 그대로** + "앱"/"서비스" 같은 키워드로 검색하고, **앱스토어를 직접** 확인한다(예: `site:play.google.com {이름}`, `site:apps.apple.com {이름}`).
   - **각도를 2~3회 바꿔** 던진다(일반 웹 + Play 스토어 + App Store). **결과가 안 나온다고 "깨끗하다"고 단정하지 않는다** — 지역·언어 한계 탓일 수 있으니, 빈약하면 검색어를 바꿔 한 번 더 확인한다.
   - **동일·유사 이름 + 유사 컨셉**이면 치명적 충돌로 보고, 발견 내용(서비스명·스토어·한 줄 설명)을 그대로 보여주며 다른 후보를 권한다.
   - 이건 **가벼운 확인**이지 정식 상표 조사가 아니다 — 발견한 것을 보고하고 최종 결정은 사용자가 한다.
4. 확정된 이름을 **확정 이름**으로 삼는다. 이후 문서는 이 이름을 쓴다.

이름이 정해지면 보이스 정의로 넘어간다.

## 보이스 정의 (질문형)

아래 질문을 **한 번에 하나씩** 던진다. 한꺼번에 늘어놓지 않는다. 답을 받으면 **짧게 받아주고 그 답 위에 쌓아서** 다음 질문으로 넘어간다. idea.md·benchmark.md를 읽었다면 그 맥락(타깃·차별 프레이밍)을 질문에 녹여 더 구체적으로 묻는다.

**예시는 그대로 쓰지 않는다 — 매번 맥락에 맞게 새로 만든다.** 각 질문에 적힌 예시는 "어떤 결의 답을 원하는지" 보여주는 **견본**일 뿐이다. 실제로 사용자에게 제시할 때는 idea.md·benchmark.md의 **도메인·타깃·컨셉에 맞춰 이 서비스에 어울리는 예시를 즉석에서 지어** 제시한다. (예: 명상 앱이면 "차분히 곁을 지키는 안내자", 운동 기록앱이면 "옆에서 같이 뛰는 친구"처럼 — 아래 견본을 복붙하지 말 것.) 좋은 예시는 사용자가 막연해하지 않고 바로 반응하게 만든다.

### 질문 1 — 브랜드 성격
"이 서비스가 사람이라면, 사람들이 뭐라고 묘사할까요?"
견본(이 서비스에 맞게 바꿔 제시): 복잡한 걸 쉽게 풀어주는 똑똑한 친구 / 모든 걸 질문하는 대담한 도전자 / 모두가 신뢰하는 차분한 전문가.

### 질문 2 — 핵심 성격 (3~5개)
"브랜드를 정의하는 형용사 3~5개를 골라주세요. 그리고 각각이 **'무엇은 아닌지'** 도 말해주세요."
견본(이 서비스에 맞게 바꿔 제시):
- "자신감 있는" — 하지만 거만하지 않은
- "캐주얼한" — 하지만 대충은 아닌
- "직설적인" — 하지만 차갑지 않은

### 질문 3 — 우리만의 금지선
"누구나 싫어하는 뻔한 말 말고, **남들은 써도 우리는 안 쓸** 표현·톤을 찾아볼게요." 라고 열고, 아래 축을 **하나씩** 던져 구체적인 회피 목록을 모은다. 한꺼번에 다 묻지 않는다. 각 축의 예시는 견본일 뿐 — idea.md·benchmark.md 맥락에 맞게 즉석에서 지어 곁들인다.

- **닮을 뻔한 톤** — "우리랑 비슷해 보이지만 살짝 어긋나서 피하고 싶은 톤이 있을까요?" (견본: 차분한 서비스인데 동기부여 코치처럼 몰아붙이는 톤)
- **강점이 과해지면** — 질문 2에서 고른 성격을 재료로 묻는다. "'친근한'이 지나치면 오지랖처럼, '자신감'이 지나치면 거만하게 들릴 수 있는데, 그중 특히 피하고 싶은 건요?"
- **이 사용자가 거슬려 하는 말투** — "우리 타깃이 유독 싫어할 만한 표현이나 말투는?" (견본: 불안한 사용자에게 다그치는 명령형, 시니어에게 영어 약어 남발)
- **경쟁이 쓰지만 우리는 안 쓸 말** *(benchmark 있을 때만)* — "차별화를 위해 일부러 안 따라갈 경쟁사 말투가 있을까요?"

목표는 "보편적으로 싫은 말"이 아니라 **이 브랜드 고유의 경계선**을 끌어내는 것. 특히 앞 두 축이 미묘한 회피를 드러낸다.

### 질문 4 — 사용자와의 관계
"누구에게 말하고, 그들이 읽고 나서 **어떤 기분이길** 바라나요?"
다음을 파고든다:
- 핵심 사용자 (누구인지, 무엇을 아는지)
- 원하는 감정 반응 (안심? 설렘? 이해받는 느낌? 자극받는 느낌?)
- 전제하는 친숙도 (처음 써보는 사람? 어느 정도 익숙한 사람?)

### 질문 5 — 피하고 싶은 보이스
'결/유형' 라벨로 묻지 않는다(여전히 추상적이다). 대신 이 서비스에서 **실제로 나올 법한 문구를 여러 톤으로 몇 줄 지어** 보여주고, **"우리는 이렇게는 말 안 했으면" 싶은 걸 고르게** 한다. **같은 상황을 톤만 바꿔** 제시하면 차이가 또렷해진다(benchmark의 결을 재료로).
- 예) *칸을 빠뜨린 날 알림* — A: "오늘도 안 하면 벌금이에요 💸" / B: "🔥 연속기록 끊겨요! 지금 안 하면 사라집니다" / C: "오늘 목표 미달성. 분발하세요."
- "이 중 우리랑 특히 **안 맞는** 톤이 있어요? 왜 그런지 한 줄만 덧붙여줘요."

### 질문 6 — 끌리는 보이스
마찬가지로 **끌릴 법한 톤의 실제 문구를 몇 줄 지어** 보여주고 **고르게** 한다.
- 예) *칸을 채운 순간* — A: "한 칸 쌓였네요. 오늘도 충분했어요." / B: "오! 오늘 것 완료 ✨" / C: "기록 완료. 좋은 흐름이에요."
- "어느 쪽이 우리답게 느껴져요? 끌리는 거 1~2개 골라주세요. (실제 동경하는 서비스가 떠오르면 그것도 좋아요.)"

### 질문 7 — 실제 문구로 톤 맞춰보기
맥락이 무엇인지 설명하지 않는다(사용자는 이미 userflow에서 화면을 다 봤다). 대신 **지금까지 답으로 잡힌 톤으로 실제 카피를 몇 줄 써서** 보여주고 반응받는다. 핵심 자리 1~2곳(예: 성공 피드백, 빈 상태)에 대해 **톤이 다른 2개 버전**을 즉석에서 만들어 제시:
- 예) 성공 피드백 — A: "오늘 칸, 채웠어요 🎉" / B: "한 칸 더 쌓였네요. 잘하고 있어요."
- "어느 쪽이 더 우리 같아요? 그리고 **특히 신경 쓰고 싶은 자리**나 **반대로 거의 안 쓸 자리**가 있으면 알려줘요." (예: "푸시는 닦달 같아 최소화", "사진 채웠을 때 피드백은 잘 해주고 싶다")

### 질문 8 — 언어 규칙 (선택)
"특정 표기 규칙이 있나요? 예: 존댓말/반말, 이모지 사용 여부와 정도, 꼭 피하고 싶은 표현, 외래어 vs 한글."
이 질문은 선택이다 — 사용자가 지쳐 보이면 건너뛰고 기본값을 추론한다.

### 질문 9 — 출력 언어
"보이스 문서는 어떤 언어로 쓸까요? (기본: 한국어)"

답을 다 모으면 Voice Profile을 생성한다.

## 산출물 — docs/brandvoice.md (필수 산출물)

**이 스킬의 최종 산출물은 항상 `docs/brandvoice.md` 파일이다.** 프로필을 채팅에만 보여주고 끝내지 않고, "파일로 저장할까요?"라고 **묻지 않는다** — 저장은 옵션이 아니라 결과물 그 자체다. 보이스 프로필 *내용*을 확인받으면(저장 여부가 아니라) 파일로 **바로 저장한다.** `docs/` 폴더가 없으면 만든다.

**언어 규칙:** 섹션 헤더와 표 컬럼명은 영어로 둬도 좋다. 본문·셀 값·설명·예시 문구는 사용자가 지정한 언어(기본 한국어)로 쓴다.

문서는 맨 위에 **확정 정보**, 그 아래 **7섹션**으로 쓴다:

```markdown
# Brand Voice — {확정 서비스 이름}

> Vibe UX · brand-voice 산출물
> 작성일: <YYYY-MM-DD>
> 기반: docs/idea.md, docs/benchmark.md

## 확정 정보
- **서비스 이름:** <확정 이름>
- **한 줄 무드:** <이 브랜드의 분위기를 한 문장으로>
```

이어서 아래 7섹션을 `##`로 단다.

### 1. Voice Summary
브랜드 보이스의 정수를 1~2문장으로. "한 줄 소개"다.
형식: `Sounds like [생생한 페르소나 묘사] who [핵심 행동/태도].`

### 2. Core Personality Traits
3~5개 성격, 각각 "We are / We are not" 틀로:
```
**[성격명]**
- We are: [실제로 어떻게 드러나는가]
- We are not: [피해야 할 오해]
- This sounds like: "[예시 문장]"
- This does NOT sound like: "[반례 문장]"
```

### 3. Tone Spectrums
6개 축에 1~5 척도로 위치를 표시한다 (●로 표시):
```
Formal    [1]——[2]——[●3]——[4]——[5]  Casual
Serious   [1]——[2]——[3]——[●4]——[5]  Playful
Authority [●1]——[2]——[3]——[4]——[5]  Peer-level
Technical [1]——[2]——[3]——[●4]——[5]  Simple
Reserved  [1]——[2]——[3]——[4]——[●5]  Enthusiastic
Bold      [1]——[●2]——[3]——[4]——[5]  Subtle
```

### 4. Vocabulary Guide
**USE** — 브랜드를 체현하는 단어/표현 10~15개 (언제/왜 쓰는지 메모와 함께)
**AVOID** — 어긋나는 단어/표현 10~15개 (왜 금지인지 메모와 함께)

### 5. Tone Adaptation Rules
보이스를 유지하면서 상황에 따라 톤을 조절하는 규칙. 제품 보이스이므로 **제품 안 맥락**을 우선한다.

**By Context (앱 안 맥락):**

| 맥락 | 톤 조절 | 예시 문구 |
|------|---------|-----------|
| 온보딩 / 첫 화면 | [조절] | "[샘플 문장]" |
| 빈 상태(empty state) | [조절] | "[샘플 문장]" |
| 성공/완료 피드백 | [조절] | "[샘플 문장]" |
| 에러/안내 | [조절] | "[샘플 문장]" |
| 푸시 알림 | [조절] | "[샘플 문장]" |
| 랜딩/소개 페이지 | [조절] | "[샘플 문장]" |

규칙: 보이스 속성은 고정, 톤은 맥락에 따라 키우거나 줄인다. 예) "bold(2)·warm(4)"인 브랜드라면 — 성공 피드백에선 warm을 5로, 에러 안내에선 bold를 1로 낮추고 warm을 올린다. 어느 속성도 사라지지 않고 균형만 바뀐다.

### 6. Style Rules
핵심 표기 규칙 5~8개.
예:
```
- 존댓말/반말: 존댓말 기본 ("~해요")
- 문장 길이: 짧고 명료하게
- 이모지: 알림·축하 맥락에서만, 최대 1개
- 숫자: 가능하면 구체 수치
- 느낌표: 한 문구에 최대 1개
```

### 7. Do's and Don'ts
구체적인 카피 작성 지침. 각 5~8개, 질문 답에서 직접 끌어온다.
**Do:** … / **Don't:** …

이미 brandvoice.md가 있으면 무엇이 바뀌는지 요약해 보여주고 확인받은 뒤 덮어쓴다.

저장 후 다음 단계를 안내한다:

> 서비스 이름과 브랜드 보이스가 정해졌어요. 이제 **to-prd**로 idea · benchmark · userflow · brandvoice 네 문서를 하나의 PRD로 묶을 수 있어요.
