Vibe UX 기획 플로우의 마지막 단계이자, 이후에도 갱신되는 살아있는 문서를 관리한다.

# To PRD

지금까지 만든 기획 산출물을 **하나의 PRD(제품 요구사항 정의서)** 로 묶는다. PRD는 단순 요약이 아니라, 이후 **vibe-build의 roadmap·plan·build 단계가 그대로 따라갈 기준 문서**다. 그래서 이 PRD는 두 얼굴을 가진다:

- **사람이 읽는 면:** 개요·배경·차별 프레이밍·브랜드처럼 "왜 이걸 만드는가"를 담는다.
- **기계가 먹는 면:** `Functional Requirements`·`User Scenarios`·`Success Criteria`·`Edge Cases`·`Assumptions` — 다음 단계(roadmap)가 마일스톤을 나누고 요구사항을 추적할 때 파싱하는 **계약 섹션**이다.

이 스킬은 Vibe UX 기획 플로우의 **마지막 단계**이고, 그다음은 vibe-build의 **roadmap**이다.

PRD는 **한 번 쓰고 끝나는 문서가 아니다.** 기획이 끝난 뒤에도 범위가 바뀌고, 기능이 추가되고, roadmap·build에서 온 피드백이 되먹임된다. 그래서 이 스킬은 **최초 통합**과 **이후 갱신** 두 모드를 가진다 — 갱신할 때는 기존 PRD를 덮어쓰지 않고 **FR/SC ID와 사람이 손댄 내용을 보존**하며 변경분만 반영한다. (FR ID의 안정성은 roadmap의 요구사항 추적이 깨지지 않게 하는 핵심이다.)

## 입력 / 산출물

입력은 작업 폴더 `docs/`의 파일들이다:

| 파일 | 만드는 스킬 | 담는 것 | 필수 |
|------|------------|---------|:---:|
| `docs/idea.md` | brainstorming-idea | 컨셉·문제·타깃·핵심 기능·플랫폼 | ✅ |
| `docs/benchmark.md` | benchmark-research | 경쟁 비교·차별 프레이밍 | ✅ |
| `docs/userflow.md` | userflow-generator | 온보딩·핵심 해피패스·화면 | ✅ |
| `docs/brandvoice.md` | brand-voice | 확정 이름·보이스·톤 | ✅ |
| `docs/DESIGN.md` | design-brandfit | 시각 디자인 시스템 | 있으면 |

산출물은 `docs/PRD.md`.

> **체인:** design-brandfit → **to-prd(PRD.md)** → vibe-build/roadmap(ROADMAP.md). roadmap은 PRD.md의 계약 섹션을 읽어 마일스톤을 나누므로, 아래 계약 섹션을 빠뜨리지 않는 것이 중요하다.

---

## Step 1 — 모드 판단 (새로 / 업데이트)

먼저 `docs/PRD.md`가 이미 있는지 확인한다.

- **없으면 → 새로 만들기 모드.** 아래 Step 2로 진행한다.
- **있으면 → 업데이트 모드.** 무엇을 할지 사용자에게 한 번 확인한다("기존 PRD를 업데이트할까요, 아니면 처음부터 새로 만들까요?"). 업데이트를 원하면:
  - **기존 `docs/PRD.md`를 먼저 읽는다.** 현재 FR/SC ID, 범위, 결정, 오픈 이슈, 사람이 손으로 추가한 내용을 파악한다.
  - **FR/SC ID를 보존한다.** 기존 ID는 절대 재배정하지 않는다. 새 요구사항은 **다음 번호로 append**(`FR-007`…). 제거되는 요구사항은 번호를 재사용하지 말고 `~~FR-004~~ (제거됨: 사유)`로 표시한다.
  - **사람이 손댄 섹션을 함부로 덮지 않는다.** 입력 문서에 없지만 PRD에만 있는 내용(직접 추가한 결정·맥락)은 변경 요청과 충돌하지 않는 한 그대로 둔다.
  - **변경분만 반영한다.** "이 기능 추가", "범위 변경", "오픈 이슈 해소" 같은 요청을 해당 섹션에만 적용하고, 상단 개정 이력에 한 줄 남긴다(최종 수정일 + 변경 요약).
  - 4개 기획 입력은 **변경의 출처가 아닐 수 있다** — 업데이트는 보통 새 결정·피드백에서 온다. 사용자가 준 변경 내용을 우선하고, 필요할 때만 입력 문서를 다시 참조한다.

## Step 2 — 입력 점검

먼저 입력 파일이 있는지 확인한다(`ls docs/` 또는 각 파일 읽기).

- **필수 4개가 다 있으면:** 각 파일을 읽고 통합으로 넘어간다. `DESIGN.md`가 있으면 함께 읽는다.
- **일부 없으면:** 어떤 파일이 없고 어느 스킬이 만드는지 알려주고, 먼저 그 스킬을 돌릴지 묻는다. 예: "userflow.md가 아직 없어요. userflow-generator를 먼저 돌릴까요? 아니면 지금 있는 것만으로 초안 PRD를 만들고 해당 섹션은 비워둘까요?"
- 사용자가 그대로 진행하길 원하면, **빠진 섹션은 `⚠️ (userflow.md 없음 — 추후 보완)`** 으로 표시하고 진행한다. 추측으로 메우지 않는다.
- `DESIGN.md`는 없어도 진행한다(아직 design-brandfit을 안 돌렸을 수 있다). 다만 9번 섹션에서 "디자인 시스템 미정"으로 표시하고, roadmap 단계 전에 design-brandfit이 필요함을 오픈 이슈에 남긴다.

파일을 읽을 때, 단순 복붙이 아니라 **서로 연결**한다 — idea의 핵심 기능이 userflow의 어느 화면/시나리오에서 실현되는지, benchmark의 차별 프레이밍이 어떤 기능으로 뒷받침되는지 머릿속에 매핑하며 읽는다.

---

## Step 3 — Functional Requirements로 정리 (PRD의 척추)

idea.md의 핵심 기능과 userflow.md의 흐름을 합쳐 **기능 요구사항 목록**을 만든다. 이게 PRD의 척추이자 roadmap이 마일스톤으로 쪼갤 단위다. 각 요구사항은:

- **안정적 ID를 갖는다 — `FR-001`, `FR-002` …** roadmap이 "어떤 FR이 어느 마일스톤에서 충족되는지" 추적 표를 만들 때 이 ID를 키로 쓴다. ID는 다시 바뀌지 않게 한다.
- **테스트 가능한 문장으로 쓴다.** "한두 줄 설명"이 아니라 검증 가능한 단언으로. 예: `FR-003: 사용자는 기록한 항목을 날짜별로 모아 볼 수 있어야 한다.` 구현 기술·파일·내부 구조는 넣지 않는다.
- **우선순위를 매긴다:**
  - **P0 (MVP, 없으면 서비스가 성립 안 됨):** 핵심 가치(aha)를 경험시키는 최소 기능. 온보딩 해피패스에 등장하는 기능은 대부분 여기.
  - **P1 (중요, 곧 필요):** 핵심 경험을 더 좋게 만들지만 첫 버전에 없어도 가치는 성립.
  - **P2 (이후):** 확장 아이디어, nice-to-have.
- **관련 시나리오/플로우를 연결한다.**

우선순위가 애매하면 사용자에게 한두 가지만 확인한다("이 기능은 첫 버전에 꼭 있어야 할까요, 아니면 다음 버전이어도 될까요?"). P0를 욕심내지 않도록 돕는다 — **MVP는 작을수록 빨리 만든다.**

---

## Step 4 — 계약 섹션 도출 (User Scenarios · Success Criteria · Edge Cases · Assumptions)

roadmap이 PRD를 기계적으로 소비하려면 아래 네 가지가 계약 형식으로 있어야 한다. FR 목록과 더불어 이걸 채운다.

- **User Scenarios** ← userflow. 핵심 흐름을 **검증 가능한 시나리오**로 푼다. 서술형 요약이 아니라, 각 시나리오에 `사용자 / 상황 / 목표 / 흐름 / 완료 조건`을 단다. "완료 조건"이 핵심 — 이 시나리오가 성공했다고 볼 수 있는 관찰 가능한 끝점.
- **Success Criteria** ← 목표에서 도출. `SC-001` ID를 달고, 가능한 한 **관찰·측정 가능한** 조건으로. 예: `SC-001: 신규 사용자가 첫 기록을 3분 안에 완료한다.` 입력 문서에 근거가 없으면 지어내지 말고, 가설임을 분명히 하거나 오픈 이슈로 남긴다.
- **Edge Cases** ← 실패 모드. 정상 흐름 밖에서 제품이 처리해야 하는 예외를 적는다. **빈 상태, 오류 상태, 중복 입력, 취소/이탈, 권한/접근 제한, 데이터 부족**을 우선 검토한다. 이건 기능을 새로 발명하는 게 아니라, 이미 정의한 플로우·기능의 **실패 지점을 짚는** 표준 작업이다. roadmap의 Milestone 1(목업)이 요구하는 "빈/로딩/오류 상태"가 여기서 나온다.
- **Assumptions** ← 입력 문서가 명확히 정하지 않았지만 PRD를 쓰기 위해 **합리적으로 정한 기본값**. 가정을 요구사항 문장 속에 숨기지 말고 여기에 분리해 적는다. (오픈 이슈와 구분: **Assumptions = 진행하려고 정한 것**, **오픈 이슈 = 아직 정해지지 않은 것**.)

---

## Step 5 — 범위 / 비범위 & 우선순위 (로드맵은 roadmap에 위임)

- **범위/비범위:** P0/P1/P2를 근거로 "첫 버전(v1)에 넣을 것"과 "지금은 안 할 것(Out of scope)"을 명시한다. 비범위를 적어두는 게 범위만큼 중요하다 — 바이브 코딩이 곁길로 새지 않게 한다.
- **우선순위 묶음(제품 관점):** P0/P1/P2를 MVP / v1 / 이후로 **제품 관점에서** 묶는다. 이건 "무엇을 먼저 만들 가치가 있는가"의 우선순위지, 구현 순서가 아니다.
- **구현 마일스톤 로드맵은 여기서 만들지 않는다.** Milestone·버티컬 슬라이스·기술 스택 같은 "어떻게/어떤 순서로 구현"은 다음 단계인 **vibe-build의 roadmap 스킬**이 `docs/ROADMAP.md`에 작성한다. PRD는 "무엇을·왜·얼마나 중요한지"까지만 책임진다. PRD 안에 별도 마일스톤 로드맵을 그리면 ROADMAP.md와 충돌하므로 그리지 않는다.

---

## Step 6 — docs/PRD.md 작성 (필수 산출물)

**이 스킬의 최종 산출물은 항상 `docs/PRD.md` 파일이다.** PRD를 채팅에만 보여주고 끝내지 않고, "파일로 저장할까요?"라고 **묻지 않는다** — 저장은 옵션이 아니라 결과물 그 자체다. 내용이 정리되면 파일로 **바로 저장한다.** `docs/` 폴더가 없으면 만든다. **업데이트 모드에서는** Step 1의 보존 규칙(FR/SC ID 유지·수동 편집 보존·번호 재사용 금지)을 지킨 병합 결과로 파일을 갱신하고, 상단 개정 이력에 한 줄을 더한다.

아래 구성으로 작성한다. **계약 섹션 5개(User Scenarios / Functional Requirements / Success Criteria / Edge Cases / Assumptions)는 roadmap이 영어 라벨로 파싱하므로 영어 제목을 그대로 둔다.** 나머지 맥락 섹션은 한글이다. 각 섹션 끝의 `←` 표시는 출처다. 날짜는 시스템이 준 오늘 날짜를 쓴다.

```markdown
# PRD — {확정 서비스 이름}

> Vibe UX · to-prd 산출물
> 작성일: <YYYY-MM-DD> · 최종 수정일: <YYYY-MM-DD>
> 개정 이력: v1 최초 통합 <YYYY-MM-DD>{업데이트 시 · v2 <변경 요약> <날짜> 한 줄씩 누적}
> 기반: idea.md, benchmark.md, userflow.md, brandvoice.md, DESIGN.md

## 1. 개요
- **한 줄 정의:** <누구를 위해, 무엇을, 어떻게>   ← idea
- **해결하는 문제:** <핵심 문제>   ← idea
- **타깃 사용자:** <핵심 사용자와 맥락>   ← idea
- **플랫폼:** <모바일 / 웹 / 반응형>   ← idea

## 2. 배경 & 근거
- **시장/경쟁 맥락:** <닮은 서비스들과 그 한계>   ← benchmark
- **우리의 차별 프레이밍:** <어디에 자리 잡고 무엇이 다른가>   ← benchmark

## 3. 목표 (Goals)
- <이 제품이 이루려는 것 1~3개>

## 4. User Scenarios   ← userflow
### Scenario 1 — <짧은 이름>
- **사용자:** <누가>
- **상황:** <어떤 맥락>
- **목표:** <달성하려는 것>
- **흐름:** <핵심 단계>
- **완료 조건:** <성공했다고 볼 수 있는 관찰 가능한 끝점>
### Scenario 2 — ...

## 5. Functional Requirements   ← idea + userflow
| ID | 요구사항 (테스트 가능한 문장) | 우선순위 | 관련 시나리오 |
|----|------------------------------|:---:|------|
| FR-001 | 사용자는 <기능/행동>을 할 수 있어야 한다. | P0 | Scenario 1 |
| FR-002 | 시스템은 <조건>에서 <기대 동작>을 제공해야 한다. | P1 | Scenario 2 |

## 6. Success Criteria
- SC-001: <대상 사용자>가 <핵심 작업>을 <조건/시간/정확도> 안에 완료할 수 있다.
- SC-002: <핵심 흐름>에서 <이탈/혼란>이 <허용 기준> 이하로 유지된다.

## 7. Edge Cases
- <예외 상황(빈 상태/오류/중복/취소·이탈/권한/데이터 부족 등)>: <기대 동작>

## 8. 화면 · 정보 구조   ← userflow
| 화면 | 목적 | 핵심 요소 |
|------|------|-----------|
| 환영 화면 | 첫 진입·가치 전달 | 가치 한 줄, [시작하기] |
| ... | ... | ... |

## 9. 브랜드 & 디자인   ← brandvoice + DESIGN
- **서비스 이름:** <확정 이름>
- **한 줄 무드:** <분위기>
- **보이스 요약:** <Voice Summary>
- **UI 카피 톤 가이드:** <Do/Don't 핵심 몇 개 — 화면 문구 작성 시 기준>
- **시각 디자인 시스템:** <DESIGN.md가 있으면 "→ docs/DESIGN.md 참조 (색·타이포·간격·컴포넌트)", 없으면 "⚠️ 미정 — roadmap 전 design-brandfit 필요">

## 10. 범위 / 비범위 & 우선순위
- **v1 범위:** <넣을 것>
- **Out of scope (지금 안 함):** <지금은 명시적으로 제외하는 것>
- **우선순위 묶음(제품 관점):** MVP <P0 기능들> / v1 <P0 + 핵심 P1> / 이후 <P1·P2·확장>
- *구현 마일스톤 로드맵은 vibe-build의 roadmap 스킬이 docs/ROADMAP.md에 작성한다.*
```

## 원칙
- **추측으로 메우지 않는다.** 입력 문서에 없는 내용은 사용자에게 묻거나 Assumptions/오픈 이슈로 남긴다. (Edge Cases의 표준 실패 모드 도출은 예외 — 이미 정의한 기능·플로우의 실패 지점을 짚는 것이지 새 기능을 만드는 게 아니다.)
- **MVP를 작게.** 모든 걸 P0로 넣으려는 충동을 막는다.
- **계약 섹션을 빠뜨리지 않는다.** Functional Requirements(ID 포함)·User Scenarios·Success Criteria·Edge Cases·Assumptions는 roadmap이 읽는 계약이다. 영어 제목·FR/SC ID를 유지한다.
- **로드맵은 roadmap에 위임.** PRD는 "무엇을·왜·우선순위"까지. "어떻게·어떤 마일스톤 순서로"는 roadmap 스킬의 몫이다.
- **PRD는 살아있는 문서.** 이미 있으면 덮어쓰지 말고 업데이트한다. FR/SC ID는 절대 재배정하지 않고, 사람이 손댄 내용을 보존하며, 변경은 개정 이력에 남긴다.
