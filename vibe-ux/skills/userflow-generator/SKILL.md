# Userflow Generator

신규 서비스를 만든다고 가정하고, **사용자가 처음 들어와서 핵심 가치를 경험하기까지의 흐름**을 그린다. 모든 진입 케이스를 분기하지 않고, **온보딩 + 핵심 해피패스(가장 잘 풀리는 정상 경로)** 에 집중하는 단순한 구조를 따른다.

이 스킬은 Vibe UX 기획 플로우의 **세 번째 단계**다. `docs/idea.md` + `docs/benchmark.md`를 입력으로 받고, `docs/userflow.md`를 산출한다. 이 산출물은 이후 PRD의 "유저 플로우 / 화면·정보 구조" 섹션으로 그대로 합쳐진다.

## 입력 / 산출물

- **입력:** `docs/idea.md`(핵심 솔루션·기능 목록·플랫폼), `docs/benchmark.md`(차별 프레이밍). benchmark가 없어도 idea만으로 진행 가능하다. idea도 없으면 "먼저 brainstorming-idea를 진행해 주세요"라고 안내하거나 사용자에게 직접 컨셉을 묻는다.
- **산출물:** `docs/userflow.md` (플로우를 하나씩 완성할 때마다 이어서 저장).

---

## 원칙

너는 앞 단계의 맥락(아이디어, 차별점)을 이미 알고 있다. 그러니 백지에서 묻기보다 **먼저 제안하고 사용자가 고르게** 한다.

**핵심 규칙:**
- **제안하되 단정하지 않는다.** 읽은 맥락으로 플로우를 제안하고, 맞는지는 사용자가 정한다.
- **사용자의 실제 순간에서 출발한다.** "사용자가 로그인 플로우에 진입한다"가 아니라 "처음 앱을 열었을 때 뭘 보면 '오 써볼까' 싶을까요?"처럼 구체적 장면으로 묻는다.
- **이상적 경험에 집중한다.** "지금 우리 시스템에서 가능한가"는 지금 고민이 아니다. 다 된다고 가정하고 어떤 경험이어야 하는지 그린다.
- **한 플로우 안에서 한 단계씩.** 전체를 한 번에 그리려 하지 말고, 첫 순간부터 차례로 짚는다.
- **선형 해피패스 위주.** 예외·분기·에러 케이스는 깊이 파지 않는다. 정상적으로 잘 풀리는 하나의 길을 또렷하게 그리는 게 먼저다. (꼭 필요한 분기만 "참고"에 한 줄로 남긴다.)

**대화 스타일:** 한국어 기본, 친근한 톤("~할까요", "~해볼게요"). 단계 전환은 대화처럼 매끄럽게.

## Phase 1: 맥락 읽고 핵심 플로우 정하기

### 1-1. 맥락을 읽는다
`docs/idea.md`와 `docs/benchmark.md`를 읽어 핵심 솔루션, 핵심 기능 목록, 차별 프레이밍, 플랫폼을 파악한다.

### 1-2. 핵심 플로우 2~3개를 제안한다
플로우는 **2~3개**로 제안한다. **1번 플로우는 온보딩**(처음 들어와 핵심 가치를 처음 경험하기까지)으로 둔다.

구성 예시:
1. **온보딩 — 처음 들어와서 핵심 기능을 처음 써보기까지** (필수)
2. **핵심 기능 해피패스 — 메인 가치를 반복적으로 쓰는 흐름**
3. (선택) **두 번째로 중요한 기능 흐름** 또는 **재방문/지속 사용 흐름**

각 플로우를 "사용자의 한마디 + 라벨"로 제시한다:

> "이거 처음인데, 뭐부터 하지?" — 온보딩
> "오늘 것도 기록해둬야지" — 핵심 기능 사용
> "지난번에 모아둔 거 다시 보고 싶다" — 재방문

사용자에게 확인한다: 이 플로우들이 맞나요? 빠진 게 있나요? 그 중 무엇부터 그려볼까요?

## Phase 2: 플로우를 화면으로 그려 논의하기

확정된 플로우를 하나씩 다룬다. 한 화면씩 따로 그리지 않는다 — **플로우 전체를 화면들로 한 번에 그려서** 흐름을 한눈에 보여주고, 그걸 놓고 논의한 뒤 **다시 그린다.** 채팅창에 ASCII 와이어프레임으로 그린다(글로만 설명하지 않는다). 한 플로우를 끝내면 **바로 저장**하고 다음으로 넘어간다.

### 진행 순서

**1. 화면 목록·순서를 먼저 합의한다 (뼈대 먼저).**
해피패스를 이루는 화면을 순서대로 제안한다: 예) "①환영 → ②습관 만들기 → ③홈 → ④기록 시트". 사용자가 순서·추가·삭제를 정하게 한다.

**2. 플로우 전체를 한 번에 그린다.**
합의된 순서대로 그 플로우의 **화면들을 ①②③④ … 쭉 이어서** 와이어프레임으로 그린다. 처음엔 러프하게. 화면 사이는 **전환 한 줄**(`[시작하기] → ②`)로 잇고, 진입부터 "핵심 가치를 경험하는 순간(aha)"까지 끊기지 않게 한다. 묻지 않고 **기본값으로 건너뛴** 결정은 주석으로 남긴다(예: "빈도·색은 기본값으로 건너뜀").

**3. 화면별 내용을 검토한다 — 여기가 기획 논의의 본체다.**
그림 아래에서 화면을 하나씩 짚으며, 각 화면에 **무슨 정보와 기능이 들어가는지**를 펼쳐 같이 검토한다. 화면마다:
- **역할** — 이 화면이 하는 **한 가지 일**.
- **담는 정보·요소** — 무엇이 보이는가, 그리고 **왜 필요한가**. 묻지 않고 기본값으로 넣은 건 표시한다(예: "로그인 없이 시작으로 그림").
- **할 수 있는 것** — 주 행동 / 보조 행동. 주 행동이 분명한지 본다.
- **정할 것** — 이 화면에 **무엇을 넣고 뺄지**, 또는 갈림길이 있을 때만 짚는다. 막연히 묻지 말고 **선택지의 장단점을 설명하고, 핵심 가치에 비춰 추천**한 뒤 사용자가 정하게 한다.
  - 예: "*연속일수(스트릭)를 홈에 띄울까?* → 동기부여엔 좋지만 첫 화면이 복잡해짐. '부담 없이'가 핵심이면 v1엔 빼고 P1로 추천."
  - "정할 것"의 기준: **핵심 경험·진입 마찰·스코프(넣을지 뺄지)·지속 사용**에 영향을 주는 결정만. 버튼 위치·색 같은 표면은 다루지 않는다. 없으면 만들지 않는다.

화면을 다 본 뒤 **흐름을 한 번** 본다: 전체에서 **뺄 화면/합칠 화면**이 있는가. "흐름이 좋은가?"가 아니라 "이 화면이 v1에 필요한가?"라는 스코프 질문으로.

**4. 합의된 결정을 반영해 다시 그린다.** 3에서 정해진 것(넣고/빼고/바꾸고)을 반영해 플로우 전체를 **업데이트해서 다시 그린다.** 필요하면 3~4를 한 번 더 반복한다. 매번 처음부터 새로 묻지 말고, 직전 그림을 기준으로 고쳐 그린다.

### 와이어프레임 그리는 규칙 (느슨하게)

- **테두리는 가볍게.** 픽셀을 맞추려 애쓰지 말 것 — 정렬은 대략만 맞으면 된다. 깔끔한 정렬보다 **무엇이 어디에 있는지 전달**이 우선이다.
- **모양은 내용에 맞게 유연하게.** 세로로 긴 화면이면 세로로, 넓은 화면이면 넓게. 정해진 틀을 강요하지 않는다.
- 안에 **핵심 요소**만 표현한다: 버튼은 `[시작하기]`, 입력은 밑줄/박스, 리스트·그리드는 기호로. 아이콘은 이모지나 기호로 대체 가능.
- **화면 이름을 일관되게** 붙인다("환영 화면", "홈", "기록 시트"). 그대로 PRD 화면 목록으로 넘어간다.

간단한 예시 (러프하게):
```
[환영 화면]
 ┌───────────────────────────┐
   ▓ ▢ ▢ ▢ ▢ ▢   (한 칸 채워지는 모션)
   매일 한 칸씩, 사진으로
                  [ 시작하기 ]
   로그인 없이 바로 시작
 └───────────────────────────┘
 → [시작하기] 누르면 ②로
```

## 플로우 작성 & 저장 — docs/userflow.md (필수 산출물)

**이 스킬의 최종 산출물은 항상 `docs/userflow.md` 파일이다.** 플로우를 채팅에만 보여주고 끝내지 않고, "파일로 저장할까요?"라고 **묻지 않는다** — 저장은 옵션이 아니라 결과물 그 자체다. `docs/` 폴더가 없으면 만든다.

각 플로우를 다 그리면 아래 구조로 정리해 보여주고, *내용*을 확인받은 뒤(저장 여부가 아니라) 파일로 **바로 저장한다.** 첫 플로우가 파일과 헤더를 만들고, 이후 플로우는 같은 파일에 이어 붙인다.

**파일 헤더 (Phase 1에서 플로우가 확정되면 먼저 작성):**

```markdown
# 유저 플로우 — {서비스 가칭}

> Vibe UX · userflow-generator 산출물
> 작성일: <YYYY-MM-DD>
> 기반: docs/idea.md, docs/benchmark.md
> 플랫폼: <모바일 / 웹 / 반응형>

## 플로우 목록
| # | 플로우 | 한 줄 요약 |
|---|--------|-----------|
| 1 | "이거 처음인데, 뭐부터 하지?" — 온보딩 | 처음 들어와 첫 성공 경험까지 |
| 2 | … | … |
```

**각 플로우 본문 (완성될 때마다 이어 붙임):** 짧은 흐름 요약 + 화면별(와이어프레임 + 한 줄 메타)로 저장한다. 채팅에서 그린 와이어프레임을 그대로 넣고, 각 화면 아래에 **역할·주요 요소·할 수 있는 것**을 한 줄씩 붙인다(논의에서 나온 내용 그대로). 이 메타가 PRD 6번(화면·정보 구조)으로 바로 넘어간다.

```markdown
---

**흐름 요약:** ①환영 → ②습관 만들기 → ③홈 → ④기록 시트

**화면**

### ① 환영 화면
\`\`\`
 ┌───────────────────────────┐
   ▓ ▢ ▢ ▢ ▢ ▢

   로그인 없이 바로 시작

\`\`\`
- **역할:** 서비스 가치를 한눈에 전하고 진입시킴
- **담는 정보·요소:** 채워지는 그리드 미리보기 · 가치 한 줄 · [시작하기]
- **할 수 있는 것:** (주) 시작하기 → ② / 로그인 없이 바로

\`\`\`

\`\`\`
- **역할:** …
- **담는 정보·요소:** …
- **할 수 있는 것:** …
- **전환:** … → ③

**핵심 가치:** <이 흐름에서 절대 놓치면 안 되는 한 가지>

**참고:** <남은 결정·분기나 설계 메모가 있으면 한두 줄로. 없으면 생략>
```
